General Coding Style Guidelines by Igor Gorlov

The conventions established by this document, as well as ones established by other documents in this repository, are strongly encouraged to be followed by every programmer working on my projects. In the meantime, all these rules are not dogmas: developers have the right to neglect or even ignore them in special cases.

Terminology

The following are some common terms used throughout the whole coding style documentation.

Spaceless case
A collective name for several styles of phrase writing in which whitespaces are not used.
Upper camel case
A spaceless case where each word of the phrase being written begins with a capital letter.
Example SpacesAreNotAllowedHere
Lower camel case
A spaceless case where each word of the phrase being written, except of the first one, begins with a capital letter.
Example spacesAreNotAllowedHere
Upper snake case
A spaceless case using an underscore character (_) as a word separator and requiring all letters to be capital.
Example SPACES_ARE_NOT_ALLOWED_HERE
Lower snake case
A spaceless case using an underscore character (_) as a word separator and totally prohibiting capital letters.
Example spaces_are_not_allowed_here
Git case
A spaceless case using hyphen character (-) as a word separator and totally prohibiting capital letters.
Example spaces-are-not-allowed-here
Archient (architectural entity)
A relatively independent part of a codebase (for example, a class or a module) which can (but is not required to) include other archients.
Archichange (architectural change)
An act of creating, deleting, renaming, replacing or relocating archients.
Archiname (architectural name)
A short Git-cased identifier of an archient.

The System of Coding Rules

This document provides a set of rules applicable to all possible projects, regardless of their type or the programming languages their are written in. Language-specific rules are defined in separate documents and, in case of overlapping, take precedence over the general guidelines; for example, "cpp.html" contains information specific to C++.

Metafiles

Metafiles are special helper documents placed at the top level of each project. In fact, they are more of a annotation of the codebase rather than its part. Only documents covered in the current section are considered metafiles; various tool-specific configs never fall into this category.

Some metafiles are mandatory, others are optional; such status is explicitly pointed out for each document (see the next subsections).

architecture.md

This mandatory metafile maps real names of architectural units to appropriate names used for commit scoping, and therefore serves as a reminder for developers. Also, "architecture.md" has an auxiliary function: it gives a quick high-level overview of the codebase and helps to understand relationships between all components.

The document must remain actual at any time and thus is required to be updated on every architectural change (that is, whenever an architectural unit appears or disappears). The update is performed by the same commit which applies the architectural change itself.

workflow.md

A project can define its own set of rules overriding both the general and the per-language instructions. Such local rules should be described by a special document called "workflow.md" which resides in the root directory of the project and is written in Markdown. The form of "workflow.md" is unspecified.

"workflow.md" is an optional metafile.

readme.md

Every project must have a "readme.md" file within its root directory. This is a Markdown document which describes the project, defines its goals and ideology, gives some basic instructions on building and/or using the product. However, "readme.md" is not a replacement for regular documentation; it only declares the essentials.

The form of "readme.md" is free except of the following:

  1. A link to the coding style guide revision used in the current repository is required.
  2. A link to "workflow.md" is required if the latter exists.

Natural Language

The official language of our community is English. It should be used in almost every case where natural language is required, including the documentation, file and directory names, identifiers within the source code, commit messages, interaction between the community members, and so on. In personal communication with me, you can also use Russian.

Project Basics

  • Name the root directory of a project in Git case.

  • Use lower snake case to name all files and directories which are (recursively) located within the root directory of a project.

  • Keep line length not higher than 120 columns.

  • Do not write source files consisting of more than 1000 lines.
