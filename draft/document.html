Document Requirements by Igor Gorlov

This guide aims to define both visual and semantic structure which, in my opinion, is optimal for the overwhelming majority of technical documents. No specific markup languages are discussed here because many of them have well-known generic subset of features (like multilevel headers and bulleted lists). However, some of the rules below involve advanced styling provided by powerful solutions (for example, HTML + CSS) but not supported in lightweight systems (such as Markdown). In the meantime, complex technologies could be a bad choice when you need to quickly write a simple document for which the contents is much more important than its visual representation. In the latter case, full-featured solutions usually produce a lot of boilerplate and hence should be given up in favor of more primitive tools. It is absolutely fair: a developer have all rights to select a method that works better on the current task. If your chosen technology doesn't implement a particular feature, feel free to ignore all rules that require this feature.

Terminology

Some terms present here may redefine (or ambiguously intersect with) existing well-known concepts. If you are in doubt, use interpretations suggested in this section.

General

Computer
Any electronic computing device.
Natural text
Text written in a natural language and bearing the main semantic load.
Computer text
Text that is one or more of the following:
  • Written in one of programming languages or in pseudocode.
  • Output of a computer program.
  • Input (including key combinations) made by a user of a computer.
  • A name of a file or a directory.
  • A URL address.
  • A VCS branch name.
Internal text
Text that is located neither within a header nor within a table of contents.
Internal link
A hyperlink referring to an element of the same document it is located in.
Typographic quotes
Double quotation marks which most important properties are described in the following table:
Glyph Hexadecimal Unicode number Decimal Unicode number HTML entity
Left (opening) double quote “ U+201C 8220 &ldquo;
Right (closing) double quote ” U+201D 8221 &rdquo;
Neutral quotes
The standard typewriter quotation marks. Both their opening and closing variants have identical glyphs ("), Unicode numbers (hexadecimal — U+0022, decimal — 34), and HTML entities (&quot;).

Partitioning

Partition
A logically complete text fragment which starts with a header and (potentially) can be nested into another partition.
Statement
A relatively small text fragment that covers an independent topic, has no header, and can not contain partitions.
Section
A top-level partition (that is, a partition that is unable to be nested). A section starts with a header of second level.
Subsection
A partition that is directly nested into a section and starts with a header of third level.
Microsection
A partition that is directly nested into a subsection and starts with a header of fourth level.
Nanosection
A partition that is directly nested into a Microsection and starts with a header of fifth level.
Picosection
A partition that is directly nested into a nanosection and starts with a header of sixth level.
Clause
A statement that is an item of a list.
Paragraph
A statement that is out of any list.

Basics

  • Begin a document with a title in the form of a top-level header (that is, a header of level one). There must be exactly one top-level header in each individual document.
  • All headers, titles, and table captions are written in header case.
  • If a header, a title or a table caption contains computer text, this computer text retains its native capitalization without any changes.
    Examples
    Mentioning a file called lib.h in a header, a wrong way Lib.h Overview
    Mentioning a file called lib.h in a header, the right way lib.h Overview
  • A large document should have a table of contents at the beginning; this is a 2-level list of internal links to sections and subsections, it has no markers preceding the elements.
  • If a table of contents is present, the document is also required to have Back to the table of contents link at the bottom.

Emphasis

Terminology

Code emphasis
An emphasis technique which uses a monospaced font and (if possible) semantically marks the text being emphasized as program code (for example, with <code> HTML tag).
Term introduction emphasis (TIE)
An emphasis technique that works on a short text fragment and makes it non-breaking italic.
Important fragment emphasis (IFE)
An emphasis technique that works on a short text fragment and makes it non-breaking bold.

The Rules of Emphasis

  • Always use a proportional font for natural text.
  • Use code emphasis for all computer text.
  • Use term introduction emphasis for the first occurrence of every new term within the internal text.
  • Use important fragment emphasis for the most valuable words and phrases.
  • TIE and IFE highlightings should not span long pieces of text (only critical fragments are to be emphasized) and should not appear too frequently.
    Examples
    An unacceptably long emphasis In our community, it is not allowed to insult other people or to impersonate the administrator.
    Too many relatively short highlightings In our community, it is not allowed to insult other people or to impersonate the administrator.
    The recommended level of emphasis In our community, it is not allowed to insult other people or to impersonate the administrator.
  • Set blue color for links, do not underline them.
  • Highlight headers of all levels with bold; the top-level header should be underlined.

Punctuation

Quotation Marks

  • Use typographic quotes if they are supported; consider neutral ones a fallback.
  • Quotation marks are placed around:
    • Direct speech.
      Example The speaker said, “Well, now look at the next slide”.
    • Inline quotes.
      Example According to Stroustrup, C++ is “the invisible foundation of everything”.
    • Titles of books, films, paintings or other artworks.
      Example Have you ever read a free book “Game Programming Patterns” by Robert Nystrom?
    • Words or phrases which are used in a figurative, non-standard way.
      Example The interpreter “knows” about all variables along with their types.
  • Quotation marks are forbidden around:
    • Computer text.
      Examples
      Sentence one, a wrong way Press “Alt + F4” to close the window.
      Sentence one, the right way Press Alt + F4 to close the window.
      Sentence two, a wrong way The new version of “iOS” has finally been released to the public.
      Sentence two, the right way The new version of iOS has finally been released to the public.
    • Organization titles.
      Examples
      Wrong “Apple” built their own closed ecosystem of products.
      Right Apple built their own closed ecosystem of products.
  • In special cases not described above, the decision about when to use quotation marks is left out for the developer.
  • If a sentence ends with both a closing quotation mark and a dot (.), the dot is placed after the closing quotation mark.
    Examples
    Wrong I thought, “We will hardly ever meet in real.”
    Right I thought, “We will hardly ever meet in real”.

Em Dashes, En Dashes, and Hyphens

  • Em dashes are used between parts of sentence (most often in place of commas or colons).
    Examples
    The subordinate clause expresses the reason of the main clause After exceeding a certain amount of threads, parallel architecture becomes a pessimization — the more workers act simultaneously, the more time the scheduler consumes.
    The clause delimited by em dashes is an optional addition for the sentence The art of programming — from my point of view — lies in the skill of working with abstractions.
  • An em dash is separated from adjacent text with a space; the space to the left of the dash must be non-breaking.
  • An en dash is used as a delimiter within ranges.
    Example Among other characters, the ASCII table includes letters (A–Z, a–z) and digits (0–9).
  • The purpose of hyphens (-) is to divide parts of compound words.
    Example C++ preprocessor allows to create both variable-like and function-like macros.
  • There must be no whitespaces directly adjacent to en dashes or hyphens.

Numerals

A number is represented digitally if one or more of the following is true:

  • It is less than zero or greater than ninety-nine.
  • It is a part of a date or a formula.
  • It is either a marker in a list or a number of a section or another number used for partitioning document.
  • It expresses a technical characteristic or a result of measurement.
If none of the above conditions are met, the number must be represented in full.
Examples
We bought twenty-five hard drives, 1 terabyte each.
4 + 8 = 12

An ordinal numeral referring to a number represented digitally must have a special ordinal suffix, which is the final two letters of the full representation of the number.

Examples
The full representation is “Two hundred and fifty-first” There is a typo on 251st page.
The full representation is “one hundred and eighteenth” Oganesson — the 118th element of the periodic system — was named after its discoverer, Russian physicist Yuri Oganessian.

Accessibility

  • Never simulate headers, lists, or any other semantic primitives using hacks and workarounds (like CSS tricks in HTML) — use these primitives directly.
  • Provide any image with a descriptive text to make graphical content understandable for screen reader users.
