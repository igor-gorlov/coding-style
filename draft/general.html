Terminology

The following are some common terms used throughout the whole coding style documentation.

Phrase Writing Methods

Spaceless case
A collective name for several styles of phrase writing in which whitespaces are not used.
Spaceful case
A collective name for several styles of phrase writing where spaces are used as word delimiters.
Header case
A spaceful case which capitalizes all notional parts of speech within the phrase being written, while the first word is unconditionally capitalized.
Example An Occurrence at Owl Creek Bridge
Upper camel case
A spaceless case where each word of the phrase being written begins with a capital letter.
Example SpacesAreNotAllowedHere
Lower camel case
A spaceless case where each word of the phrase being written except of the first one begins with a capital letter.
Example spacesAreNotAllowedHere
Upper snake case
A spaceless case using an underscore character (_) as a word separator and requiring all letters to be capital.
Example SPACES_ARE_NOT_ALLOWED_HERE
Lower snake case
A spaceless case using an underscore character (_) as a word separator and totally prohibiting capital letters.
Example spaces_are_not_allowed_here
Git case
A spaceless case using hyphen character (-) as a word separator and totally prohibiting capital letters.
Example spaces-are-not-allowed-here

Miscellaneous

Line terminator
A character intended to finish a line of text, also widely known as end-of-line, newline or linebreak (for example, CRLF on Windows or LF on Unix-compatible systems).
Compound word
A word consisting of two or more hyphen-separated parts.
To represent (a number) digitally
To write a number using digits.
Examples
Arabic decimal 126
Hexadecimal 0x7e
Roman CXXVII
To represent (a number) in full
To write a number using words.
Example One hundred and twenty-six

Project Basics

  • Encode all files within a repository in UTF-8.
  • Finish any file with a line terminator.
  • Name the root directory of a project in Git case.
  • Use lower snake case to name all files and directories which are (recursively) located within the root directory of a project.
  • Keep line length not higher than 120 columns.
  • Do not write source files consisting of more than 1000 lines.
  • It is recommended to store all external dependencies of a project inside its own repository, in third_party folder, with one subfolder per each dependency.
  • To simplify creation of new files with initial content or insertion of frequently repeated code fragments into existing files, you can make a templates catalog within the root directory of your project and store samples there.

Metafiles

Metafiles are special helper documents written in Markdown and placed at the top level of each project. In fact, they are more of a annotation of the codebase rather than its part. Only documents covered in the current section are considered metafiles; various tool-specific configs never fall into this category.

Some metafiles are mandatory, others are optional. See the next subsections for details.

workflow.md

A project can define its own set of rules overriding both the general and the per-language instructions. Such local rules are described by an optional metafile called workflow.md, which has an unspecified form.

readme.md

Every project must have a readme.md file within its root directory. This document describes the project, defines its goals and ideology, gives some basic instructions on building and/or using the product. However, readme.md is not a replacement for regular documentation; it only declares the essentials.

The form of readme.md is free except of the following:

  1. A link to the coding style guide revision used in the current repository is required.
  2. A link to workflow.md is required if the latter exists.
