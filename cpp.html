C++ Coding Style Guide by Igor Gorlov

The guidelines present in this document are strongly encouraged to be followed by every C++ programmer working on my projects. In the meantime, these guidelines are only advises, not strict rules: developers have the right to neglect or even ignore them in special cases.

General

  • Use C++17.

  • Do not use global variables.

  • Use #pragma once in order to prevent multiple inclusions of single file. Do not use conditional compilation for this purpose.

  • Use struct keyword for plain old data types.

Naming

  • Use informative names for all entities except of temporary variables.

    Examples
    An important variable 
    uint64 lastModificationTimestamp;
    A temporary variable used only as a loop counter 
    uint i = 0;

  • Give include files .h extension.

  • Give .cpp extension to files that should be compiled directly to binaries.

  • Use upper camel case for namespace and type names (except of aliases for arithmetic types, which should be written in lower camel case).

    Examples
    A user-defined namespace 
    namespace LibraryPrefix
    {
    	// Declarations and definitions...
    }
    A user-declared type 
    class CustomType;
    An arithmetic type alias 
    typedef std::uint32_t uint32;

  • Use lower camel case to name functions and variables.

    Examples
    A function call 
    saveCurrentFile();
    A variable declaration 
    int playerPosition;

  • Use upper snake case for preprocessor identifiers.

    Example 
    #define NUMBER_OF_ITERATIONS

  • Use lower snake case to name source files and directories containing source files.

    Example 
    path_to/source_file.cpp

  • Add m prefix to names of private class members.

    Example 
    class Allocator
    {
    private:
    	void* mAllocatedMemory;
    };

  • Add p prefix to names of protected class members.

    Example 
    class Grandfather
    {
    protected:
    	std::string pFamilySkeleton;
    };

  • Prefer verbs for function names and nouns for type, namespace and variable names.

    Examples
    A good function name 
    void deallocateMemory(void* memory);
    An undesirable function name 
    void memoryDeallocation(void* memory);
    An appropriate variable name 
    enemy* target;
    An inconvenient variable name 
    enemy* destroy;

  • Bool variables, as well as functions returning bool, may have names in the form of yes-no question.

    Examples
    A function 
    bool isRunning(ProcessHandler* process);
    A data member of a class 
    class Player
    {
    public:
    	bool isAlive;
    };

  • If a variable stores amount of some items, give it a name with numberOf prefix; but if such a variable contains an actual list of items, use simple plural.

    Example 
    std::vector<std::string> messages;
    messages.push_back("Alarm!");
    messages.push_back("Alarm canceled");
    uint64 numberOfMessages = messages.size();

Formatting

  • Organize each source file according to the following template:

    1. An optional multiline comment briefly describing this file and its contents, followed by two empty lines.

    2. An optional #pragma once, followed by two empty lines.

    3. An optional sequence of standard includes in alphabetical order, followed by one empty line.

    4. An optional sequence of system includes in alphabetical order, followed by one empty line.

    5. An optional sequence of third-party includes in alphabetical order, followed by one empty line.

    6. An optional sequence of includes from current project in alphabetical order, followed by one empty line.

    7. A sequence of declarations and definitions, optionally divided into paragraphs by one or two empty lines.

  • Keep line length not higher than 120 columns.

  • Do not write source files consisting of more than 1000 lines.

  • Place each curly brace, which encloses a scope, on a separate line.

    Example 
    if(condition)
    {
    	doSomething();
    	doSomethingElse();
    }

  • If a pair of curly braces is empty, place both these braces on one separate line without any whitespaces between them.

    Example 
    class ObjectWithEmptyConstructor
    {
    public:
    	ObjectWithEmptyConstructor()
    	{}
    };

  • Always use curly braces to enclose code blocks, even when the language rules permit not to use them in this particular situation.

    Examples
    A dirty trick allowed by C++ 
    if(condition)
    	thenExecuteSingleStatement();
    The more reliable way 
    if(condition)
    {
    	thenExecuteSingleStatement();
    }

Indentation

  • For indentation use tabs (one per level), not spaces.

  • Do not indent access specifiers.

    Example 
    class Container
    {
    public:
    	void* externalData;
    private:
    	void* mInternalData;
    };

  • Indent comments in the exactly same manner as code.

Commenting

  • When finishing a class or namespace body which occupies a lot of vertical space, put a single-line comment right after the closing curly brace (right after semicolon in case of class). The mentioned comment should be equivalent to the header of this class or namespace.

    Examples
    A namespace 
    namespace Prefix
    {
    	// Many declarations and definitions...
    } // namespace Prefix
    A class 
    class Fake
    {
    	// A huge amount of members...
    }; // class Fake

  • If a single-line comment (//...) does not fit on one line, use a multiline variant (/*...*/) instead.

  • Try to comment on any code that is not self-documented.

  • Finish any comment with a period.
